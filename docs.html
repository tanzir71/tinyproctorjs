Implementation guide

Everything you need to add browser integrity monitoring to your exam or quiz platform: one script tag on the exam page, one endpoint on your server.

Quick start

Two steps, however you work: get the file and add the tag — then give your server a place to receive events. Pick whichever path fits you.

Path A — ask your AI agent (recommended)

Using Claude, Copilot, Cursor or any coding agent? Paste this prompt and it will wire everything into your stack, including the collector:

Add tinyproctor.js exam integrity monitoring to my project:
1. Fetch https://tanzir71.github.io/tinyproctorjs/tinyproctor.min.js
   and add it to my static assets.
2. Add its script tag to my exam/quiz page with data-endpoint pointing
   at a new /collect endpoint, plus my real exam and attempt IDs.
3. Implement /collect for my backend following
   https://tanzir71.github.io/tinyproctorjs/llms.txt — store events with
   attemptId, violationKey and integrityScore so I can review attempts.

The full machine-readable integration contract lives at /llms.txt — payload schema, config reference, and validation rules in one plain-text file any agent can read.

Path B — one command

The library is a single static file. Download it straight into your project's assets:

curl -LO https://tanzir71.github.io/tinyproctorjs/tinyproctor.min.js

Want the repo with the demo server and test suite too? git clone https://github.com/tanzir71/tinyproctorjs

Path C — manual

1. Host the script. Serve tinyproctor.min.js (~6 KB gzipped) from your own domain. No CDN, no third-party calls.

2. Embed it on the exam page.

<script
  src="/js/tinyproctor.min.js"
  data-tinyproctor
  data-endpoint="https://your-domain.com/collect.php"
  data-exam-id="FINAL_2026_06"
  data-attempt-id="ATTEMPT_12345"
  data-candidate-label="student_42">
</script>

3. Receive events. The script batches events and POSTs JSON to your data-endpoint. A complete collector is ~30 lines — copy-paste the PHP or Node.js examples below.

Monitoring starts automatically as soon as the script loads — no init call needed.

Every signal can be disabled individually. Clipboard and context-menu actions can also be blocked (not just detected) with data-block-clipboard and data-block-contextmenu.

A heartbeat event fires every 30 seconds (configurable) so your backend can detect dropped sessions and confirm the monitor stayed alive for the full attempt.

Integrity scoring

The score starts at 100 and decreases with each weighted violation, clamped to [0, 100]:

score = 100 − Σ ( weight[key] × count[key] )

Weights are fully configurable. A practical reading of the default scale:

  • 80–100 — clean or near-clean session
  • 50–79 — worth a human review
  • 0–49 — multiple or severe flags; investigate
Important: the score is a triage signal, not proof of cheating. A dropped window focus can be a notification popup; a paste can be legitimate. Always pair automated flags with human review before any academic-integrity decision.

JavaScript API

For programmatic control, disable auto-init and configure in JS:

<script src="/js/tinyproctor.min.js" data-tinyproctor data-autoinit="false"></script>
<script>
  var proctor = TinyProctor.init({
    endpoint: '/collect',
    examId: 'FINAL_2026_06',
    attemptId: attempt.id,
    candidateLabel: user.internalId,
    weights: { fullscreen_exit: 30 },
    onViolation: function (key, data, state) {
      // e.g. warn the candidate in your own UI
      if (state.integrityScore < 50) showWarningBanner()
    },
    onEvent: function (payload) {
      // every event (heartbeats included), before it is sent
    }
  })

  // Enforce fullscreen at exam start (must be called from a user gesture)
  startButton.addEventListener('click', function () {
    proctor.requestFullscreen()
  })
</script>

Payload schema

Events are sent as JSON batches via navigator.sendBeacon() (on page exit) or fetch(..., { keepalive: true }):

{
  "schemaVersion": 1,
  "clientVersion": "1.0.0",
  "kind": "batch",
  "sessionId": "tp_18f3a...",
  "examId": "FINAL_2026_06",
  "attemptId": "ATTEMPT_12345",
  "candidateLabel": "student_42",
  "sentAt": "2026-06-12T10:30:00.000Z",
  "userAgent": "Mozilla/5.0 ...",
  "language": "en-US",
  "screen": { "w": 1920, "h": 1080 },
  "timezoneOffsetMin": -360,
  "events": [
    {
      "schemaVersion": 1,
      "clientVersion": "1.0.0",
      "clientTime": "2026-06-12T10:29:58.512Z",
      "eventType": "visibility",
      "violationKey": "tab_hidden",
      "eventData": { "visibilityState": "hidden" },
      "pageUrl": "https://exams.school.edu/final",
      "visibilityState": "hidden",
      "hasFocus": false,
      "isFullscreen": false,
      "violationCount": 3,
      "integrityScore": 75
    }
  ]
}

eventType is one of heartbeat, visibility, focus, fullscreen, devtools, clipboard, contextmenu, print, idle, custom. Informational events (e.g. focus regained) have violationKey: null.

Delivery is at-least-once: failed batches are re-queued with exponential backoff, so treat (sessionId, clientTime, eventType, violationKey) as a deduplication key if exact counts matter.

Building a collector

The collector is the only server-side piece: an endpoint that accepts JSON batches and stores them. Any stack works.

PHP + SQLite (single file)

<?php // collect.php
header('Access-Control-Allow-Origin: https://exams.school.edu');
header('Access-Control-Allow-Methods: POST, OPTIONS');
header('Access-Control-Allow-Headers: Content-Type');
if ($_SERVER['REQUEST_METHOD'] === 'OPTIONS') { http_response_code(204); exit; }
if ($_SERVER['REQUEST_METHOD'] !== 'POST') { http_response_code(405); exit; }

$raw = file_get_contents('php://input');
// support data-format="form" too:
if (isset($_POST['payload'])) $raw = $_POST['payload'];
$batch = json_decode($raw, true);
if (!$batch || ($batch['kind'] ?? '') !== 'batch') { http_response_code(400); exit; }

$db = new PDO('sqlite:' . __DIR__ . '/events.sqlite');
$db->exec('CREATE TABLE IF NOT EXISTS events (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  session_id TEXT, exam_id TEXT, attempt_id TEXT, candidate TEXT,
  client_time TEXT, event_type TEXT, violation_key TEXT,
  integrity_score REAL, violation_count INTEGER, data TEXT,
  received_at TEXT DEFAULT CURRENT_TIMESTAMP
)');
$stmt = $db->prepare('INSERT INTO events
  (session_id, exam_id, attempt_id, candidate, client_time, event_type,
   violation_key, integrity_score, violation_count, data)
  VALUES (?,?,?,?,?,?,?,?,?,?)');
foreach (($batch['events'] ?? []) as $e) {
  $stmt->execute([
    $batch['sessionId'] ?? '', $batch['examId'] ?? '', $batch['attemptId'] ?? '',
    $batch['candidateLabel'] ?? '', $e['clientTime'] ?? '', $e['eventType'] ?? '',
    $e['violationKey'] ?? null, $e['integrityScore'] ?? null,
    $e['violationCount'] ?? 0, json_encode($e['eventData'] ?? null)
  ]);
}
http_response_code(204);

Node.js (no dependencies)

// collect.mjs — node collect.mjs
import { createServer } from 'node:http'
import { appendFileSync } from 'node:fs'

const ORIGIN = 'https://exams.school.edu'

createServer((req, res) => {
  res.setHeader('Access-Control-Allow-Origin', ORIGIN)
  res.setHeader('Access-Control-Allow-Methods', 'POST, OPTIONS')
  res.setHeader('Access-Control-Allow-Headers', 'Content-Type')
  if (req.method === 'OPTIONS') { res.writeHead(204).end(); return }
  if (req.method !== 'POST' || req.url !== '/collect') { res.writeHead(404).end(); return }

  let body = ''
  req.on('data', (c) => { body += c; if (body.length > 1e6) req.destroy() })
  req.on('end', () => {
    try {
      const batch = JSON.parse(body)
      if (batch.kind !== 'batch') throw new Error('bad payload')
      appendFileSync('events.ndjson', JSON.stringify(batch) + '\n')
      res.writeHead(204).end()
    } catch {
      res.writeHead(400).end()
    }
  })
}).listen(8787)
Production checklist: validate attemptId against an active attempt in your database, rate-limit per session, cap request body size, and store the raw batch for audit. Reject events for attempts that are already submitted or expired.

CORS

If the exam page and collector are on different origins, the collector must respond to OPTIONS preflight and send Access-Control-Allow-Origin (see examples above). Notes:

  • navigator.sendBeacon() sends no custom headers and skips preflight when the content type is simple — both collector examples accept it as-is.
  • For cookie-based auth across origins, set data-credentials="include" and return Access-Control-Allow-Credentials: true with an explicit (non-wildcard) origin.
  • Same-origin deployments (recommended) need no CORS configuration at all.

LMS & platform integration

Moodle

Add the script tag to your quiz page via a theme layout override, the "Additional HTML" admin setting (footer), or a local plugin that injects it only on mod/quiz attempt pages. Pass Moodle's attempt ID into data-attempt-id.

WordPress (LearnDash, Tutor LMS, Quiz Maker)

Enqueue the script conditionally on quiz pages with wp_enqueue_script, then print the data attributes with a script_loader_tag filter — or simply paste the tag into your quiz template.

Custom platforms

Render the script tag server-side with the attempt's real IDs. Generate data-attempt-id when the attempt starts and verify it server-side when events arrive.

Privacy & honest limits

What it never collects: no camera, no microphone, no screen recording, no keystroke contents, no clipboard contents. Events describe that something happened (e.g. "paste occurred in a textarea"), never what was pasted.

Self-hosted by design: the script and the data both live on your infrastructure. No third-party processors, which dramatically simplifies GDPR/FERPA conversations.

Honest limits: client-side JavaScript cannot see other monitors, other devices, or a phone next to the keyboard, and a technically skilled candidate can disable any in-page script. tinyproctor.js is a deterrence and triage layer for low/medium-stakes assessments — for high-stakes certification, combine it with stronger controls. FAQ

Does it work on mobile?

Yes — visibility, focus, clipboard, idle, and heartbeat signals all work on mobile browsers. Fullscreen and DevTools signals are desktop-oriented.

Will it slow the exam page down?

No. It's ~6 KB gzipped, dependency-free, uses passive event listeners and batched network calls (one small POST every ~0.9s at most).

What happens if the network drops?

Events queue locally (up to queueMax) and retry with exponential backoff. On page close, the queue is flushed via sendBeacon, which browsers deliver even during unload.

Can candidates fake a perfect score?

A sufficiently technical candidate can — this is true of every client-side tool. Use heartbeat gaps and missing-session detection server-side to flag tampering: a session with no events at all is itself a red flag.
